## Question

What should the product do when it sent an open request but cannot tell whether
the controller opened the compartment?

## Context

The legacy client returned the assignment to `Available` after its final timeout.
A timeout proves only that the client did not receive a result. The controller
may have opened the door and lost its response. Releasing the assignment would
let a later claim start while the first door action remained possible.

When the product holds the claim, it sends some recipients to a site operator
even when the door never opened. When the product releases the claim, it reduces
operator work but can overlap two pickup sessions and make parcel custody
unclear.

## Decision

The Parcel Operations Product Owner and Locker Safety Owner chose to retain the
claim and enter `Recovery required` until the controller definitively reports
the original request's result and door state.

## Consequences

- A second pickup claim cannot overlap a possible door open.
- A recipient may need operator help after a harmless network failure.
- Site operators need a scoped reconciliation action and clear audit events.
- The product must reuse one request ID so the controller can recognize the
  original action.

## Alternatives considered

### Release immediately after timeout

- Rejected because a lost success response can leave the door open while another
  claim begins.
- Reconsider when the controller contract can supply a definitive non-execution
  result despite a client timeout.

### Issue a new open request automatically

- Rejected because a new request ID may cause a second hardware action and hides
  whether the first request executed.
- Reconsider when the hardware contract defines a higher-level pickup session ID
  that deduplicates both requests.

### Mark the pickup collected

- Rejected because a timeout does not prove an open or subsequent close.
- No planned reconsideration.
